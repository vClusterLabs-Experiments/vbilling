Private Node Billing

How vBilling detects and bills dedicated bare-metal or virtual machines allocated to individual vCluster tenants. Ideal for neoclouds, GPU clusters, and high-isolation environments.

What Are Private Nodes?

Private nodes are dedicated Kubernetes nodes (physical servers, cloud VMs, or bare-metal machines) allocated exclusively to a single vCluster tenant. Unlike the default shared model where vCluster pods run alongside other workloads on shared nodes, private nodes give a tenant the entire machine.

Shared Mode (default)

vCluster pods run in a namespace on shared nodes. Billing is based on actual pod resource usage (CPU, memory) as reported by metrics-server.

Private Node Mode

Entire nodes are dedicated to one tenant. Billing is based on the full node capacity (CPU, memory, GPUs, storage) regardless of actual utilization.

Common use cases for private nodes:

  • GPU workloads -- Tenants get dedicated GPU nodes (e.g., 8x A100 or 8x H100)
  • Security isolation -- Tenants require physical machine-level isolation
  • Performance -- No noisy-neighbor effects from shared infrastructure
  • Bare-metal providers -- Neoclouds selling entire servers to tenants

How vBilling Detects Private Nodes

vBilling discovers private nodes by searching for Kubernetes nodes with specific labels that associate them with a vCluster. The detection uses multiple strategies in order:

Strategy 1: Direct namespace match

vBilling searches for nodes with these label selectors (where the value matches the vCluster namespace):

# Label selectors tried for namespace "vcluster-team-gpu": vcluster.loft.sh/managed-by=vcluster-team-gpu vcluster.loft.sh/cluster=vcluster-team-gpu

Strategy 2: Name with prefix stripping

If the namespace starts with a common prefix (vcluster- or vc-), vBilling also tries the name with the prefix stripped:

# For namespace "vcluster-team-gpu", the prefix "vcluster-" is stripped: # vCluster name derived = "team-gpu" # Additional selectors tried: vcluster.loft.sh/managed-by=team-gpu vcluster.loft.sh/cluster=team-gpu
!
This prefix stripping is a common source of label mismatch issues. If your namespace is vcluster-team-gpu and you label nodes with vcluster.loft.sh/managed-by=vcluster-team-gpu, it will match via Strategy 1. But if you label with just team-gpu, it matches via Strategy 2. See Troubleshooting for details.

Strategy 3: Custom label via environment variable

For non-standard setups, you can specify a custom label selector via the VCLUSTER_NODE_LABEL environment variable:

# Format: VCLUSTER_NODE_LABEL=label-key=value-pattern # Use %s as a placeholder for the vCluster name (after prefix stripping) # Example: nodes labeled with "tenant=team-gpu" export VCLUSTER_NODE_LABEL="tenant=%s" # Example: nodes labeled with "dedicated-to=team-gpu" export VCLUSTER_NODE_LABEL="dedicated-to=%s" # In Helm values: # env: # - name: VCLUSTER_NODE_LABEL # value: "tenant=%s"

What Gets Billed

When private nodes are detected for a vCluster, the full node capacity is billed to the tenant. The following data is extracted from each node:

ResourceSourceLago Metric
CPU (total cores) node.status.capacity["cpu"] vcluster_cpu_core_hours (billing_mode=private_node)
Memory (total bytes) node.status.capacity["memory"] vcluster_memory_gb_hours (billing_mode=private_node)
GPU count + type node.status.capacity["nvidia.com/gpu"] + node labels vcluster_gpu_hours (billing_mode=private_node)
Ephemeral storage node.status.capacity["ephemeral-storage"] Tracked internally (not a separate Lago metric)
Node count Number of matched nodes vcluster_private_node_hours
Spot/on-demand status Node labels (see below) Used to determine IsSpot flag
Instance type node.kubernetes.io/instance-type Tracked in InstanceType field

Additionally, vBilling queries metrics-server for the actual CPU and memory usage on private nodes (via node-level metrics) and adds that to the billing totals. This ensures workloads running directly on dedicated nodes outside the vCluster namespace are also captured.

GPU Type Detection on Private Nodes

vBilling checks multiple node labels to determine the GPU model on a private node. Labels are checked in priority order -- the first match wins:

LabelPlatformExample Value
nvidia.com/gpu.product NVIDIA GPU Operator (any platform) NVIDIA-A100-SXM4-80GB
nvidia.com/gpu.machine Alternative NVIDIA label NVIDIA-DGX-A100
cloud.google.com/gke-accelerator Google Kubernetes Engine (GKE) nvidia-tesla-t4
k8s.amazonaws.com/accelerator Amazon Elastic Kubernetes Service (EKS) nvidia-tesla-a100

GPU type strings are normalized: spaces are replaced with hyphens and leading/trailing whitespace is trimmed.

Spot / On-Demand Detection

Each private node is checked for spot/preemptible status using the following labels:

LabelSpot ValuesPlatform
kubernetes.io/lifecycle "spot", "preemptible" Common / generic
node.kubernetes.io/lifecycle "spot", "preemptible" Alternative generic
cloud.google.com/gke-preemptible "true" GKE preemptible VMs
cloud.google.com/gke-spot "true" GKE Spot VMs
eks.amazonaws.com/capacityType "SPOT" Amazon EKS
karpenter.sh/capacity-type "spot" Karpenter autoscaler

The spot/on-demand status is stored in the IsSpot field of each private node. This information can be used for pricing differentiation.

Example: GKE Test with T4 GPU

Here is a real-world example of setting up private node billing on GKE with NVIDIA T4 GPUs.

Label the node

# Identify the GPU node kubectl get nodes -l cloud.google.com/gke-accelerator=nvidia-tesla-t4 # NAME STATUS AGE # gke-cluster-gpu-pool-abc12345-def0 Ready 2d # Label the node as belonging to the vCluster kubectl label node gke-cluster-gpu-pool-abc12345-def0 \ vcluster.loft.sh/managed-by=team-gpu

Verify detection

# Check that the node has the right labels kubectl get node gke-cluster-gpu-pool-abc12345-def0 --show-labels # Labels include: # cloud.google.com/gke-accelerator=nvidia-tesla-t4 # vcluster.loft.sh/managed-by=team-gpu # node.kubernetes.io/instance-type=n1-standard-8 # Check vBilling logs (namespace: vcluster-team-gpu) kubectl logs -n vbilling-system -l app=vbilling -f # Expected output: # [metrics] vcluster-team-gpu: found 1 private node(s): totalCPU=8 cores, totalMemory=30.00 GB, totalGPUs=1 # [metrics] node=gke-cluster-gpu-pool-abc12345-def0 type=n1-standard-8 cpu=8 mem=30720Mi gpus=1(nvidia-tesla-t4) on-demand # [metrics] vcluster-team-gpu: added private node usage: CPU=+2.150 cores, Memory=+8.45 GB # [controller] sent 6 billing events to Lago

Resulting Lago events

With a 60-second collection interval (0.01667 hours), the following events are sent:

# CPU core-hours from private node (8 cores x 0.01667h) { "code": "vcluster_cpu_core_hours", "properties": { "cpu_core_hours": 0.133333, "raw_cpu_cores": 8.0, "billing_mode": "private_node" } } # Memory GB-hours from private node (30 GB x 0.01667h) { "code": "vcluster_memory_gb_hours", "properties": { "memory_gb_hours": 0.5, "raw_memory_gb": 30.0, "billing_mode": "private_node" } } # GPU hours (1 T4 x 0.01667h) { "code": "vcluster_gpu_hours", "properties": { "gpu_hours": 0.016667, "gpu_count": 1, "gpu_type": "nvidia-tesla-t4", "billing_mode": "private_node" } } # Private node hours (1 node x 0.01667h) { "code": "vcluster_private_node_hours", "properties": { "private_node_hours": 0.016667, "node_count": 1, "billing_mode": "private_node" } }

How to Set Up Private Nodes

Follow these steps to enable private node billing for a vCluster:

1. Label your nodes

# Option A: Use the standard label kubectl label node <node-name> \ vcluster.loft.sh/managed-by=<vcluster-name> # Option B: Use the cluster label kubectl label node <node-name> \ vcluster.loft.sh/cluster=<vcluster-name> # For multiple nodes: kubectl label node node-1 node-2 node-3 \ vcluster.loft.sh/managed-by=team-gpu

2. Ensure the label value matches

!
The label value must match what vBilling expects. vBilling tries several values:
  • The full namespace name (e.g., vcluster-team-gpu)
  • The namespace with vcluster- or vc- prefix stripped (e.g., team-gpu)
  • The custom label pattern from VCLUSTER_NODE_LABEL

3. Verify with kubectl

# Check which nodes match the label kubectl get nodes -l vcluster.loft.sh/managed-by=team-gpu # NAME STATUS ROLES AGE VERSION # gpu-node-001 Ready <none> 5d v1.29.2 # gpu-node-002 Ready <none> 5d v1.29.2 # Inspect node details kubectl describe node gpu-node-001 | grep -A5 "Capacity:" # Capacity: # cpu: 96 # ephemeral-storage: 1844592Ki # hugepages-1Gi: 0 # memory: 1056718848Ki # nvidia.com/gpu: 8

4. Check vBilling logs

kubectl logs -n vbilling-system -l app=vbilling | grep "private node" # [metrics] vcluster-team-gpu: found 2 private node(s): totalCPU=192 cores, totalMemory=2013.12 GB, totalGPUs=16 # [metrics] node=gpu-node-001 type=p4d.24xlarge cpu=96 mem=1006559Mi gpus=8(NVIDIA-A100-SXM4-80GB) on-demand # [metrics] node=gpu-node-002 type=p4d.24xlarge cpu=96 mem=1006559Mi gpus=8(NVIDIA-A100-SXM4-80GB) on-demand
✓
Private node billing happens automatically alongside shared-namespace billing. Both shared and private metrics are sent to Lago simultaneously. The billing_mode=private_node property lets you distinguish them in Lago.
